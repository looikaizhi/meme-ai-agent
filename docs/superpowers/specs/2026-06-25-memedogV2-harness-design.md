# memedogV2 执行外壳设计规格

## 一、背景

当前 `memedogV2` 已经形成了从地址输入、确定性硬过滤、证据采集、看多看空审计、最终裁决到交易信号的管线。现有实现中，硬过滤部分主要由确定性代码控制，但审计阶段仍然依赖提示词要求 Codex 使用 GMGN 相关能力。这个方式存在一个核心问题：即使某一次观察到 Codex 使用了技能，也不能证明下一次仍然会按同样的流程执行。

因此，本设计的目标不是继续增强提示词，而是引入一个可验证、可回放、可替换大模型后端的执行外壳。执行外壳负责控制流程、调用工具、记录证据和校验结果；大模型只在被允许的步骤中做结构化推理。

## 二、目标

1. 生产路径不再依赖大模型是否自觉调用技能。
2. 执行外壳必须显式控制 GMGN 数据抓取、硬过滤、证据整理、看多分析、看空分析和最终裁决。
3. 同一套输入可以在 Codex 和其他大模型接口厂商之间对比。
4. 每次运行都生成结构化运行记录，用于审计、回放和问题定位。
5. 保留对 Codex 是否真实遵守工作流的评测能力，但不把这种不确定行为作为生产主路径。
6. GMGN 技能需要有可复现的版本来源，避免环境漂移影响评测结果。

## 三、非目标

1. 本设计不改动现有 `src/memedog/`。
2. 本设计不要求马上接入真实交易执行。
3. 本设计不把大模型工具调用能力作为唯一数据来源。
4. 本设计不把所有大模型厂商都模拟成可执行本地工具的代理。普通大模型只参与结构化推理。
5. 本设计不在第一阶段实现完整仪表盘，只保留运行记录供后续展示。

## 四、总体方案

采用混合方案：

1. 生产主路径使用确定性执行外壳。
2. 评测路径使用代理合规执行外壳。
3. 两条路径共享输入、证据格式、输出格式、运行记录和校验规则。

生产主路径中，执行外壳直接调用 `gmgn-cli` 或夹具数据，构造证据包，然后把证据包交给大模型做看多、看空和裁决。大模型不能决定是否抓取 GMGN 数据，也不能决定跳过哪一个必要步骤。

评测路径中，执行外壳把完整任务交给 Codex 这类可执行代理，并检查它是否真的按要求完成步骤。该路径用于评估 Codex 或其他代理的工作流遵守能力，不作为生产信号的唯一来源。

## 五、目录设计

建议新增以下目录：

```text
src/memedogV2/harness/
├── contracts.py
├── workflow.py
├── tool_registry.py
├── model_registry.py
├── recorder.py
├── runner.py
├── compliance.py
└── replay.py
```

各文件职责如下：

`contracts.py` 定义执行外壳运行记录、步骤结果、工具调用记录、模型调用记录和合规检查结果。

`workflow.py` 定义固定工作流状态机。第一阶段的状态顺序是：读取安全数据、读取信息数据、执行硬过滤、构建证据包、看多分析、看空分析、最终裁决、生成信号。

`tool_registry.py` 定义工具适配层。第一阶段至少支持真实 GMGN 命令、夹具数据和伪造工具三种来源。

`model_registry.py` 定义大模型适配层。第一阶段至少支持 Codex 后端和伪造模型后端；其他接口厂商通过同一个结构化调用接口接入。

`recorder.py` 负责记录每一步的输入摘要、输出摘要、耗时、错误、工具调用证据和模型调用证据。

`runner.py` 实现生产主路径的确定性执行。

`compliance.py` 实现代理合规评测。它检查代理是否完成必要步骤、是否输出合法结构、是否存在缺失步骤、是否引用了不存在的证据。

`replay.py` 从历史运行记录或夹具数据回放，不触发真实网络请求。

## 六、生产主路径

生产主路径如下：

```text
地址输入
-> 执行外壳运行器
-> 读取 GMGN 安全数据
-> 读取 GMGN 信息数据
-> 硬过滤
-> 构建证据包
-> 看多模型调用
-> 看空模型调用
-> 裁决模型调用
-> 交易信号
-> 运行记录
```

在这个路径中，GMGN 数据读取由执行外壳发起，不由大模型发起。硬过滤继续使用现有 `HardFilter` 和字段映射规则。证据包应优先从已经抓取的 GMGN 原始数据中提取，只有确实需要补充维度时，才允许执行外壳调用额外工具。

大模型调用只接收结构化证据包和固定角色指令。看多模型只能输出看多观点；看空模型只能输出风险观点；裁决模型只能输出最终信号。每次模型调用都必须有严格输出结构。

## 七、代理合规评测路径

代理合规评测路径如下：

```text
评测任务
-> 代理后端
-> 代理尝试自行执行工作流
-> 收集代理输出和可见证据
-> 合规检查器
-> 合规报告
```

Codex 这类可执行代理可以被要求使用技能或命令完成任务。执行外壳需要检查它是否满足以下条件：

1. 是否产生了必要步骤的结构化结果。
2. 是否按预期顺序完成步骤。
3. 是否有工具调用证据或等价输出证据。
4. 是否使用了合法的字段来源。
5. 是否输出符合结构要求的最终结果。
6. 是否在缺失数据时显式降级，而不是编造数据。

普通大模型接口厂商不应被要求直接调用本地技能。对它们的评测重点是：在给定同一份证据包时，是否能稳定输出符合结构要求的看多、看空和裁决结果。

## 八、核心数据契约

执行外壳需要新增以下概念。

`HarnessRun` 表示一次完整运行，包含运行编号、地址、后端名称、运行模式、步骤列表、最终信号和合规结果。

`StepResult` 表示一个步骤的结果，包含步骤名称、状态、输入引用、输出引用、工具调用记录、模型调用记录、开始时间、结束时间和错误信息。

`ToolCallRecord` 表示一次工具调用，包含工具名称、命令摘要、输入摘要、输出摘要、退出状态和耗时。

`ModelCallRecord` 表示一次模型调用，包含模型后端、角色、输入引用、输出引用、结构校验结果和耗时。

`ComplianceResult` 表示合规检查结果，包含是否通过、已观察步骤、缺失步骤、结构是否有效、工具证据是否有效和违规列表。

## 九、工作流状态机

第一阶段固定工作流如下：

1. `读取安全数据`：调用 GMGN 安全数据命令或夹具。
2. `读取信息数据`：调用 GMGN 信息数据命令或夹具。
3. `硬过滤`：使用现有规则判断是否淘汰。
4. `构建证据包`：从安全数据、信息数据和硬过滤结果中抽取大模型需要的证据。
5. `看多分析`：模型只输出看多论点。
6. `看空分析`：模型只输出风险论点。
7. `最终裁决`：模型综合证据和双方论点，输出信号。
8. `记录结果`：保存运行记录，支持后续回放和对比。

如果硬过滤淘汰，后续模型步骤应标记为跳过，而不是继续调用模型。

如果数据源失败，根据配置选择失败关闭或带标记放行。无论哪种策略，都必须写入运行记录。

## 十、大模型后端设计

大模型后端分为两类。

第一类是结构化推理后端。它只接收输入并返回结构化结果，不具备工具执行权。其他大模型接口厂商应归入这一类。

第二类是可执行代理后端。它可以被要求运行命令或调用技能。Codex 属于这一类。可执行代理后端只能用于合规评测或受控实验，不应成为生产主路径中 GMGN 数据抓取的唯一方式。

两类后端都必须通过统一接口接入。统一接口至少包含：角色名称、输入文本、输出结构、工作目录、超时和运行记录钩子。

## 十一、GMGN 技能保存策略

建议保存 GMGN 技能的版本信息，但不要把它作为生产主路径的唯一依赖。

第一阶段推荐保存：

```text
tools/gmgn-skills.lock.json
scripts/install_gmgn_skills.sh
```

锁文件记录技能来源、提交版本、安装时间、期望命令和校验摘要。安装脚本负责把指定版本安装到本地 Codex 环境。

如果后续需要完全离线或完全可复现的代理评测，再考虑增加：

```text
third_party/gmgn-skills/
```

生产主路径仍然应该优先依赖 `gmgn-cli` 命令契约、字段映射、夹具和执行外壳工具适配层。GMGN 技能主要用于测试 Codex 是否能按技能工作流执行。

## 十二、运行记录与回放

每次执行外壳运行都应保存一份结构化记录。记录位置建议为：

```text
runs/memedogV2/
```

运行记录文件名建议包含日期、运行编号和地址摘要。记录中不要保存敏感密钥。原始响应可以保存到引用文件中，主记录只保存摘要和引用路径。

回放模式必须满足：

1. 不触发真实网络请求。
2. 不调用真实大模型。
3. 使用历史输入和夹具输出复现硬过滤、证据构建和信号生成流程。
4. 能比较不同模型后端在同一证据包下的输出差异。

## 十三、测试策略

第一层是纯单元测试。覆盖数据契约、状态机、记录器、合规检查器和回放器。

第二层是夹具集成测试。使用现有 `tests/memedogV2/fixtures` 中的 GMGN 响应，验证执行外壳可以从地址走到硬过滤和信号生成。

第三层是伪造模型测试。通过伪造模型后端返回固定看多、看空和裁决结果，验证生产主路径不依赖真实大模型。

第四层是 Codex 手动冒烟测试。验证 Codex 后端能否在合规评测模式中按要求完成工作流。

第五层是跨模型对比测试。在相同证据包下比较不同大模型接口厂商的输出结构、推荐结果、置信度和理由。

## 十四、错误处理

GMGN 命令返回限流时，执行外壳不得循环重试。它应记录限流状态，并根据配置跳过、暂停或返回降级结果。

工具输出无法解析时，当前步骤标记失败，运行记录保留原始错误摘要。

模型输出不符合结构时，该模型步骤标记失败。生产主路径可以选择重试一次，但重试也必须记录。

代理合规评测中，如果代理缺少步骤、编造证据、输出结构无效或无法证明工具使用，则合规结果应失败。

## 十五、分阶段实施建议

第一阶段只实现确定性执行外壳主路径，不接入真实跨厂商模型。目标是让生产路径不再依赖模型是否调用技能。

第二阶段实现运行记录和回放，确保每次信号都可审计。

第三阶段实现 Codex 合规评测，专门回答“Codex 是否真的按工作流执行”。

第四阶段接入其他大模型接口厂商，做同证据包对比。

第五阶段根据评测结果决定是否保留、镜像或版本锁定完整 GMGN 技能目录。

## 十六、验收标准

1. 使用夹具数据时，执行外壳能稳定生成相同运行记录。
2. 硬过滤淘汰时，不会调用看多、看空或裁决模型。
3. 给定同一个证据包，不同模型后端可以通过统一接口运行。
4. 每个步骤都有可审计的输入引用、输出引用和状态。
5. Codex 合规评测可以明确报告缺失步骤和结构违规。
6. 生产主路径即使不安装 GMGN 技能，也可以通过 `gmgn-cli` 和夹具运行。
7. GMGN 技能的版本来源有锁定记录，便于复现实验。

## 十七、主要风险

第一，Codex 的底层执行记录可能不可完全获取。如果无法获取完整工具调用记录，合规评测需要依赖输出证据、文件产物和步骤报告进行间接判断。

第二，不同大模型对同一证据包的风险偏好会不同。因此比较结果不能只看是否推荐，还要记录置信度、证据引用和理由。

第三，GMGN 字段可能变化。字段映射仍然需要集中维护，并通过夹具测试发现变化。

第四，保存完整 GMGN 技能目录会增加维护成本。除非需要完全离线复现，否则第一阶段只锁定来源和版本。

## 十八、结论

推荐采用确定性执行外壳作为生产主路径，并把 Codex 技能调用能力放到代理合规评测路径中。这样可以同时解决两个问题：生产信号不再依赖大模型是否自觉调用技能，同时仍然能够测试 Codex 和其他大模型后端在相同任务下的行为差异。

GMGN 技能需要被版本锁定，并在必要时保存到目录中，但不应该成为生产主路径的唯一可信执行机制。真正可信的部分应是执行外壳状态机、工具适配层、结构化证据、运行记录和可回放测试。

## 十九、落地状态(2026-06-25,Phase 1 + 运行记录 已实现)

生产主路径已落地于 `src/memedogV2/harness/`,并经真实环境验证:

- **确定性取数**:`tool_registry.py`(`GmgnCliToolSource` 复用带限速/缓存/429 的 `GmgnCli`,`FixtureToolSource` 供离线测试),每次抓取产 `ToolCallRecord`。GMGN 数据由外壳发起,大模型不参与抓取。
- **确定性证据**:`evidence_builder.build_evidence(facts, ca)` 从已抓取的 `token info` facts 经 `FIELD_MAP` 抽取(smart money/KOL/dev 战绩);`dev_graduation_rate` 无 gmgn 来源,恒 None 并进 `missing`。**审计阶段零额外 gmgn 调用、零 LLM 抓数**。
- **可换后端**:`model_registry.py` 统一接口 `complete(*, role, prompt, schema) -> (dict, ModelCallRecord)`。`DeepSeekBackend`(OpenAI 兼容 API,`json_object` + 一次修复重试)、`CodexBackend`(strict `--output-schema`)、`FakeBackend`(单测)。`__main__` 默认 `deepseek`,可传 `codex`。
- **固定工作流**:`runner.py` 顺序 read_security→read_info→hardfilter→build_evidence→bull→bear→judge→signal;硬过滤淘汰则模型步骤标记 SKIPPED;`run()` 永不抛(后端报错/裁决畸形 → FAILED 步 + 无信号)。
- **运行记录**:`recorder.py` 写 `runs/memedogV2/*.json`(每步 + 工具/模型调用证据;`runs/` 已 gitignore)。
- **真实环境测试**:`tests/memedogV2/live/`(`-m live`,默认不跑),覆盖真实 gmgn-cli、真实 DeepSeek、真实 Codex、真实全链路。已实测通过。

**仍延后(本次未做)**:`compliance.py`(§七 代理合规评测)、完整 `replay.py` 跨模型回放、`tools/gmgn-skills.lock.json`(§十一,生产已不依赖技能,留待合规评测阶段)。
